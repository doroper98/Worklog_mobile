## 1. 30초 브리핑

Workwiki Mobile은 데스크탑 Workwiki의 **모바일 보조 클라이언트(PWA)**.
주된 용도는 세 가지: **① 이동 중 위키·일지 조회, ② 회의 중 빠른 메모를
inbox로 전송, ③ 보고 있는 문서에 대해 Claude 앱과 연결**. 자체 AI 채팅은
없다. claude.ai에 위임한다.

주 사용자는 단 한 명이다. 단일 사용자 도구이며, 로그인·온보딩·협업 기능은
없다. **정보 밀도와 재진입 속도**가 UX의 핵심 지표다.

---

## 2. 사용자 프로필

한 명만 존재한다. 기술 친숙도 높음. UI 설명·튜토리얼 불필요. 오히려 텍스트
대비 정보 밀도를 높이는 편이 이 사용자에게 이롭다. 한국어 주 사용, 영어
기술 용어 혼재. iPhone 16+, Android 둘 다 사용.

---

## 3. 핵심 사용 시나리오

### S1. 회의 중 생각을 붙잡는다 (Quick Memo)
회의 중 아이디어가 떠오른다. 폰을 꺼내 PWA를 열고, 하단 FAB을 탭하고, 텍스트
혹은 음성으로 메모하고, 전송한다. **화면 전환 최소, 탭 최소, 대기 시간 최소.**
총 소요 시간 15초 이하.

### S2. 이동 중 문서를 읽는다 (Read)
출퇴근 지하철, 점심 후 카페. 어제 데스크탑에서 쓴 일지를 다시 읽거나,
특정 프로젝트 위키를 검토한다. **한글 장문의 편안한 가독성**이 핵심.

### S3. 문서에 대해 Claude와 대화한다 (Bridge)
읽던 문서에서 궁금한 점이 생긴다. 화면 하단의 "💬 Claude에 묻기" 버튼을
탭하면 OS 공유 시트가 뜨고, "Claude" 앱을 선택하면 문서 내용이 새 대화
입력창에 로드된다. **Workwiki Mobile의 역할은 여기서 끝난다.**

## 4. 화면 목록 (총 10개)

핸드오프 번들에 각 화면의 HTML·CSS·React 구조가 포함되어 있다. 아래는 **맥락·목적**만 정리.

### 4.1 AuthGate
최초 접속 시 GitHub PAT 입력. "왜 PAT이 필요한가"를 접어두고 한 탭으로 펼칠 수 있어야 함.

### 4.2 HomeView
최상위 진입. 오늘·위키·최근·보낸 메모로 빠른 진입. **재진입 속도가 체감 품질을 결정**.

### 4.3 WikiCategory
People/Projects/Issues/Notes 중 하나의 리스트 뷰.

### 4.4 MarkdownView
문서 열람. **한글 장문 가독성이 최우선.** 행간·단락 간격·본문 폭 충분히. wikilink는 내부 링크로. 하단 고정 액션 바(뒤로 / Claude에 묻기 / 북마크).

### 4.5 SearchView
전문 검색. 패싯 필터, 하이라이트.

### 4.6 QuickMemoSheet
빠른 메모 bottom sheet. **iOS/Android 키보드 회피 필수.** 전송 후 즉시 닫히고 원래 화면 복귀. 확인 다이얼로그 없음.

### 4.7 InboxSent
보낸 메모 히스토리. 데스크탑에서 처리된 메모는 "✓ 병합됨" 배지.

### 4.8 SettingsView
PAT 관리, **테마(라이트/다크/시스템 자동)**, 본문 글자 크기, 로컬 캐시 관리.

### 4.9 ShareOutcome
"Claude에 묻기" 실행 피드백. **존재감을 최소화**. 공유 시트는 네이티브가 제공.

### 4.10 OfflineBanner
글로벌 오프라인 알림 배너. 눈에 띄되 주요 UI 가리지 않음.

## 5. 글로벌 UI 요구사항

### 5.1 라이트/다크 모드 — 필수
- 세 옵션: 라이트 / 다크 / **시스템 자동**(기본값)
- Settings에서 수동 오버라이드 가능
- 둘 다 동등한 품질로 디자인됨
- 전환은 즉시 반영, 부드러운 애니메이션
- OS가 다크/라이트를 바꾸는 순간에도 즉시 반영

### 5.2 한글 가독성
- 본문 장문 읽기가 흔함
- 한글에 최적화된 행간, 글자 간격, 폰트 크기
- 사용자 조절 가능한 본문 크기 (소/중/대)

### 5.3 터치 UX
- 터치 타깃 최소 44×44pt (iOS) / 48dp (Android)
- FAB은 엄지 닿는 범위에 배치
- 스와이프는 OS 기본 외에 최소화
- 롱프레스는 컨텍스트 메뉴에만

### 5.4 피드백
- 비동기 작업 로딩 표시. 즉각성 중요한 곳은 스켈레톤 UI
- 성공 토스트 3초 자동 사라짐
- 에러는 원인과 다음 행동 한 줄로

### 5.5 접근성
- VoiceOver/TalkBack 지원
- Dynamic Type 대응
- WCAG AA 이상 대비 (라이트·다크 모두)
- 아이콘 + 라벨 병기

## 6. 디자인 톤

### 6.1 분위기 (Brief 규정 — 번들이 이 원칙 위에서 구체화)
- **생산성 도구의 차분함.** 시각 노이즈 최소화.
- **오래 봐도 피곤하지 않음.** 화려한 그라데이션, 강한 채도, 애니메이션 남발 지양.
- **정보 밀도는 허용.** 사용자가 개발자 수준.
- **"조용한 유능함"** 이 키워드.

### 6.2 피해야 할 것
- 인스타그램·SNS 앱의 활기찬 분위기
- 소비자 엔터테인먼트 앱의 풍부한 컬러
- 과도한 일러스트, 마스코트, 친근한 카피
- 의미 없는 장식 요소

### 6.3 Claude Design이 확정한 것
구체적 색상 팔레트, 타이포그래피, 간격 체계, 아이콘 스타일, 애니메이션은
핸드오프 번들에서 확정됨. Claude Code는 번들을 fetch하여 그대로 구현할 것.

**원칙에서 벗어난 결정이 번들에 있다면** — 예를 들어 강한 채도 컬러나
과도한 애니메이션이 있다면 — Claude Code는 사용자에게 확인 후 진행한다.

## 7. 기술 제약

### 7.1 플랫폼
- iOS Safari 16+, Android Chrome 최신
- PWA standalone 모드
- 홈화면 추가 후 네이티브처럼 실행
- 스플래시 스크린과 아이콘(192, 512, 512-maskable)

### 7.2 오프라인
- Service Worker로 앱 셸과 최근 문서 캐시
- 오프라인 읽기 가능
- 쓰기는 IndexedDB 큐 후 연결 복원 시 자동 전송

### 7.3 성능
- Lighthouse PWA 점수 90+
- 첫 인터랙션까지 3초 이내
- 스크롤 60fps 유지

## 9. 디자인하지 않는 것

- AI 채팅 UI — claude.ai 위임
- 복잡한 마크다운 편집기 — 빠른 메모만
- 협업·공유 기능 — 단일 사용자
- 푸시 알림 — 인앱 토스트만
- 프로필·아바타 — 한 명뿐
- 온보딩 투어 — PAT 입력 안내 외 없음
- 다국어 — 한국어 단일

## 11. 성공 기준

디자인이 성공했다면 다음이 모두 참이어야 한다.

1. 사용자가 15초 안에 메모를 저장하고 원래 하던 일로 복귀할 수 있다.
2. 한글 장문 일지를 10분 이상 읽어도 눈이 피로하지 않다.
3. 밤늦게 누운 상태에서 다크 모드로 열어도 눈부시지 않다.
4. 낮 야외에서 라이트 모드로 열어도 글씨가 잘 보인다.
5. 홈화면 아이콘만 보고도 "이건 Workwiki 모바일"임을 즉시 알아본다.
6. 공유 시트에서 Claude 앱으로 넘어가는 경험이 자연스러워 앱 전환을 의식하지 못한다.
7. 첫 사용 시 튜토리얼 없이 주요 기능을 발견할 수 있다.

## 12. 함께 주는 맥락

Workwiki Mobile의 디자인 언어가 데스크탑 Workwiki와 완전히 일치할 필요는
없지만 **연속성은 있어야 한다.** 데스크탑은 Electron · Windows 전용 ·
Tokyo Night 팔레트 계열 · 시스템 폰트 · 정보 밀도 높음.

모바일은 이것을 그대로 복제할 필요 없다. "같은 도구의 모바일 버전"이라는
느슨한 가족 유사성이면 충분.
